\begin{abstract}
    \vspace{4pt}
Laporan ini mendokumentasikan transisi bot trading BTC-QUANT dari simulasi ke
\emph{live trading} di Lighter mainnet. Selama periode 10--27 Maret 2026, sistem melewati tiga fase:
eksekusi manual (10--13 Mar), uji coba otomasi (15 Mar), dan eksekusi penuh otomatis
(16--27 Mar). Dari total 27 perdagangan yang dicatat, 26 perdagangan valid
(ekskluding anomali bug) menghasilkan win rate \textbf{73.1\%}, total PnL \textbf{+\$4.83},
dan expected value \textbf{+\$0.186 per trade} dengan profit factor 2.03. Dua belas
perbaikan kritikal diimplementasikan --- mencakup deteksi exit OCO, autentikasi token SDK,
sinkronisasi posisi startup, dan optimasi parameter strategi HestonStrategy.
Hasil menunjukkan bahwa win rate tinggi (73--80\%) mampu mengkompensasi risk-reward
ratio kurang dari 1 secara matematis, menghasilkan positive expectancy yang sustainable.

\vspace{4pt}
\noindent\textbf{Keywords:}
algorithmic trading; decentralized exchange; perpetual futures; risk management; order management system
\end{abstract}

\subsection{Metodologi dan Fase Eksekusi}
\label{subsec:methodology}

Periode live trading dibagi menjadi tiga fase dengan karakteristik eksekusi berbeda:

\begin{itemize}
    \item \textbf{Fase Manual (10--13 Mar):} Sinyal dari bot dibaca, order dibuka
          dan ditutup secara manual melalui dashboard Lighter. Intervensi manual
          untuk menggeser TP/SL umum dilakukan pada fase ini.
    \item \textbf{Uji Coba Otomasi (15 Mar):} Test run awal otomasi bot. Fase ini
          tidak dihitung dalam statistik performa karena bersifat diagnostik.
    \item \textbf{Fase Otomatis (16--27 Mar):} Eksekusi penuh oleh bot. Intervensi
          manual sesekali masih dilakukan untuk menggeser target TP/SL.
\end{itemize}

    \item EV \textbf{$+\$0.186$/trade} bermakna positive expectancy — strategi

    \item Profit factor \textbf{2.03} mengindikasikan 2 unit gross profit untuk

    \item R:R ratio 0.75 (di bawah 1) dikompensasi oleh win rate tinggi:
          \[ \text{EV} = (0.731 \times 0.501) - (0.269 \times 0.670) = +0.186 > 0 \]
    \item Fase otomatis (80\% WR, EV $+\$0.278$) membuktikan konsistensi eksekusi
          bot melampaui eksekusi manual.
\end{itemize}

\subsection{Detail Perbaikan Kritikal}

\subsubsection{FIX-1: Double Position Guard}
\textbf{Commit:} \texttt{492cf0b} | \textbf{Severity:} \textcolor{darkred}{CRITICAL}

\paragraph{Masalah.} Bot dapat membuka posisi baru saat posisi sebelumnya masih \texttt{OPEN} di exchange, mengakibatkan \textit{double exposure} yang tidak terkontrol.
\paragraph{Solusi.} Implementasi \texttt{get\_open\_position()} \emph{check} sebelum setiap entry. Jika terdapat posisi aktif, sinyal baru akan di-skip.
\paragraph{Dampak.} Mencegah risiko \textit{margin call} tidak terduga akibat penumpukan posisi.

\subsubsection{FIX-2: Signal Replay setelah Restart}
\textbf{Commit:} \texttt{d8e7a87} | \textbf{Severity:} HIGH

\paragraph{Masalah.} Setelah restart, sinyal lama yang sudah diproses dapat di-replay karena \texttt{last\_notified\_ts} hanya tersimpan di memori (\textit{volatile}).
\paragraph{Solusi.} Persistensi \texttt{last\_notified\_ts} ke file \texttt{last\_candle\_ts\allowbreak{}.json} untuk menjaga state melewati restart.
\paragraph{Dampak.} Menghilangkan entri duplikat yang tidak valid setelah bot melakukan \textit{reboot}.

\subsubsection{FIX-3: SL Order Authentication}
\textbf{Commit:} \texttt{c3107f9} | \textbf{Severity:} \textcolor{darkred}{CRITICAL}

\paragraph{Masalah.} Stop-loss order gagal ditempatkan karena \texttt{\_submit\_order()} tidak menyertakan \textit{auth header} yang valid.
\paragraph{Solusi.} Seluruh \textit{order placement request} dikonfigurasi menggunakan \textit{auth token} yang valid di header HTTP.
\paragraph{Dampak.} Menjamin proteksi posisi. Kegagalan ini menyebabkan kerugian \$6.30 pada 18 Maret.

\subsubsection{FIX-4: Advisory Signal Tidak Tereksekusi}
\textbf{Commit:} \texttt{3160dc1} | \textbf{Severity:} MEDIUM

\paragraph{Masalah.} Sinyal dengan conviction 15--20\% (\texttt{ADVISORY}) terblokir oleh logika filter meskipun merupakan sinyal valid.
\paragraph{Solusi.} Pembaruan \textit{logic gate} untuk memperlakukan status \texttt{ADVISORY} setara dengan \texttt{ACTIVE}.
\paragraph{Dampak.} Meningkatkan frekuensi trading dengan menangkap peluang pada rentang \textit{conviction} menengah.

\subsubsection{FIX-5: Lighter Auth Token via SDK}
\textbf{Commit:} \texttt{e28c312} | \textbf{Severity:} \textcolor{darkred}{CRITICAL}

\paragraph{Masalah.} Format token manual tidak valid; Lighter memerlukan tanda tangan EdDSA via \textit{private key}.
\paragraph{Solusi.} Menggunakan SDK \texttt{create\_auth\_token\_with\_expiry()} untuk menghasilkan token resmi yang ditandatangani.
\paragraph{Dampak.} Memulihkan akses ke seluruh \textit{write operations} dan \textit{private queries} yang sebelumnya \textit{unauthorized}.

\subsubsection{FIX-6: Exit Price dari \texttt{trigger\_price}}
\textbf{Commit:} \texttt{e28c312} | \textbf{Severity:} MEDIUM

\paragraph{Masalah.} \texttt{filled\_price} bernilai 0.0 pada order SL/TP karena mekanisme \textit{trigger order} di Lighter.
\paragraph{Solusi.} Implementasi logika \textit{fallback} untuk mengambil harga eksekusi dari field \texttt{trigger\_price} jika \texttt{filled\_price} kosong.
\paragraph{Dampak.} Akurasi kalkulasi PnL dan data \textit{risk management} menjadi 100\% valid.

\subsubsection{FIX-7: Startup Position Sync}
\textbf{Commit:} \texttt{861822f} | \textbf{Severity:} MEDIUM

\paragraph{Masalah.} Ketidaksinkronan status posisi jika posisi ditutup saat bot sedang \textit{offline}, menyebabkan bot terjebak dalam status \texttt{OPEN}.
\paragraph{Solusi.} Penambahan fungsi \texttt{sync\_position\_status()} saat \textit{daemon startup} untuk rekonsiliasi state dengan exchange.
\paragraph{Dampak.} Bot selalu memulai dengan \textit{state} yang bersih dan akurat setelah \textit{downtime}.

\subsubsection{FIX-8: Exit Type Detection}
\textbf{Commit:} \texttt{861822f} | \textbf{Severity:} MEDIUM

\paragraph{Masalah.} Penentuan jenis exit (SL vs TP) berbasis \textit{price distance} tidak reliabel saat terjadi \textit{slippage} tinggi.
\paragraph{Solusi.} Identifikasi eksplisit menggunakan field \texttt{order\_type} dari API response (\textit{stop} vs \textit{take-profit}).
\paragraph{Dampak.} Logika \textit{SL~freeze} dan pelabelan \textit{risk manager} menjadi lebih akurat.

\subsubsection{FIX-9: L2 EMA Weakening Logic}
\textbf{Commit:} \texttt{8ab9b1b} | \textbf{Severity:} LOW

\paragraph{Masalah.} Filter tren (L2 EMA) tidak cukup kuat memblokir sinyal kontra-tren; hanya mengurangi \textit{confidence} tanpa membatalkan sinyal.
\paragraph{Solusi.} Memaksa status \texttt{l2\,=\,False} saat kondisi \textit{weakening} terdeteksi, sehingga sinyal otomatis di-skip.
\paragraph{Dampak.} Mengurangi \textit{false entry} pada kondisi pasar yang mengalami pembalikan tren.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\subsubsection{FIX-10: OCO Canceled Order Filter}
\textbf{Commit:} \texttt{f644601} | \textbf{Severity:} \textcolor{darkred}{CRITICAL}

\paragraph{Masalah.} Saat TP tercapai, SL order yang dibatalkan otomatis (OCO) terbaca sebagai eksekusi SL, memicu \textit{freeze} palsu.
\paragraph{Solusi.} Filter eksplisit untuk mengabaikan order berstatus \texttt{"cancel"} dalam pemrosesan riwayat transaksi.
\paragraph{Dampak.} Mencegah pemblokiran entri (\textit{freeze}) yang keliru setelah perdagangan untung (TP).

\subsubsection{FIX-11: Manual SL Freeze Clear}
\textbf{Tanggal:} 27 Mar | \textbf{Severity:} MEDIUM

\paragraph{Masalah.} Status \textit{freeze} yang tersimpan di disk memblokir entri selama berjam-jam meskipun masalah teknis sudah teratasi.
\paragraph{Solusi.} Pembersihan manual pada \texttt{sl\_freeze\_state\allowbreak{}.json} dan \textit{restart} kontainer untuk mereset kondisi bot.
\paragraph{Dampak.} Mengurangi waktu \textit{idle} bot dan mempercepat pemulihan operasional setelah kegagalan beruntun.

\subsubsection{FIX-12: Dynamic Boot Log}
\textbf{Commit:} \texttt{62a32b0} | \textbf{Severity:} LOW

\paragraph{Masalah.} Log \textit{startup} menampilkan nilai \textit{margin} statis yang menyesatkan dan tidak sesuai konfigurasi aktual.
\paragraph{Solusi.} Sinkronisasi log dengan variabel \texttt{MARGIN\_USD} dan \texttt{LEVERAGE} yang aktif secara dinamis.
\paragraph{Dampak.} Memudahkan \textit{monitoring} dan \textit{debugging} konfigurasi strategi saat bot dijalankan.

\subsection{[FEAT-1] Switch ke HestonStrategy}
\label{subsec:heston}

\noindent\textbf{Commit:} \texttt{01ef707} \quad \textbf{Tanggal:} 26 Mar

\paragraph{Alasan Switch.}
FixedStrategy menggunakan SL/TP tetap (1.333\%/0.71\%) yang tidak adaptif terhadap
kondisi volatilitas pasar. HestonStrategy menggunakan parameter ATR-adaptive:
\begin{itemize}
    \item SL = ATR $\times$ 1.5 ($\approx$2.2\% dari entry)
    \item TP = ATR $\times$ 2.1 ($\approx$3.1\% dari entry)
    \item R:R ratio $\approx$ 1:1.4
\end{itemize}

\section{Kesimpulan}
\label{sec:conclusion}

Periode 10--27 Maret 2026 membuktikan viabilitas BTC-QUANT sebagai sistem trading
otomatis pada decentralized perpetual exchange. Empat pencapaian utama dicatat:

\begin{enumerate}
    \item \textbf{Live Trading Aktif:} 27 perdagangan dieksekusi di Lighter mainnet
          dengan sistem yang stabil setelah 12 perbaikan kritikal.
    \item \textbf{Positive Expectancy Tervalidasi:} EV $+\$0.186$/trade, win rate
          73.1\%, dan profit factor 2.03 pada 26 perdagangan valid.
    \item \textbf{Keunggulan Otomasi:} Fase otomatis (80\% WR, EV $+\$0.278$)
          mengungguli eksekusi manual secara konsisten.
    \item \textbf{Strategy Upgrade Justified:} HestonStrategy terbukti superior
          ($+46$ pp PnL, Sharpe $+1.1$) dibanding FixedStrategy dalam backtest 62 hari.
\end{enumerate}

\paragraph{Key Learnings.}
\begin{itemize}
    \item Auth token untuk DEX memerlukan kriptografik signing, bukan format manual.
    \item OCO order harus difilter berdasarkan status \texttt{"cancel"} sebelum
          digunakan untuk menentukan jenis exit.
    \item Gunakan field \texttt{order\_type} eksplisit dari API untuk deteksi SL/TP,
          bukan distance heuristics.
    \item Startup reconciliation penting untuk menjaga konsistensi state posisi.
\end{itemize}

\paragraph{Roadmap.}
Langkah berikutnya mencakup implementasi PR-2 (partial TP + break-even SL),
pengembangan adaptive position sizing berbasis drawdown, dan monitoring performa
parameter \$20$\times$7x pada 10 trade pertama sebelum scaling ke \$30$\times$5x.
